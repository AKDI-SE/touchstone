§ 1 · 为什么做这件事

PR 评审要规模化，有四个绕不开的难点

信号噪声

AI 评审噪声是乘性的

多来源并行，假阳性叠加成数倍噪声面；噪声一大，开发者会无视全部意见，评审从增益变负担。

成本

每个 PR 跑全量验证付不起

真正判断"改得对不对"要测试与验证，但对每个 PR 都跑全套既慢又贵，无法落到日常流水线。

问责

谁为"合入"负责

把合并按钮交给 AI，出问题谁背？需要一条清晰、可审计的边界，区分"AI 能自动放行"与"必须人来点"。

可信

"测试绿"不等于"对"

author 自己写的验收测试可能绕过真实意图；需要独立、盲于实现的验证，否则自动化只是自欺。

§ 2 · 方案总览

顾问式评审 + 复用开源 + 三层防护

定位

默认顾问式

只产高信噪比的评审意见与风险分流，不产准入；合入按钮始终在人手里。自动合并与验证可选、默认关。

不造轮子

评审复用 PR-Agent

评审广度交给成熟开源工具；本系统只写 PR-Agent 没有的：归一、裁决映射、契约与栈规则、验证、自治、校准。

边界

自主边界 = 验证边界

能自动放行多少，由客观验证覆盖到哪决定——不由评审 agent 的数量或模型多样性决定。

  • 向下守底线：客观、可复现的质量门禁是唯一准入，判断层全错也越不过它。
  • 向上提上限：AI 判断扩展评审广度、降低盲点，但只给建议。
  • 离线回灌：用真实的人审裁决与合入结果，反过来校准判断、据结果收回或给予自主权。

§ 4 · 关键机制

四个把"边界"落到代码的机制

判断

三类判定，不同处置

正确性（疑问→验证）/ 红线·契约（规则化→机器判定）/ 质量（建议）。category 决定风险等级与处置路径。

分流

风险三档验证

低/中→cheap_only；高→targeted_tests；高且影响面严重→full_suite（多跑变异）。把贵的验证只花在最危险的改动上。

验证

独立验收 + 充分性校验

异模型、盲于实现地生成验收测试；改动覆盖→空实现哨兵→变异，逐级确认"测得够不够"。规格须人核准。

自治

多道门禁串联 + 熔断

自动合并需：统一门禁绿 · 无阻断否决 · 闭环收敛 · 变更分类达标 · 总开关开。合入后异常越阈→熔断收回自主权。

§ 6 · 演进与已知风险

走到哪了，还剩什么风险

演进主线

  • 主干设计：三类判定 / 风险分流 / 契约 / 校准 / 固化 / 熔断。
  • 验证子系统：独立验收 + 充分性校验，双 runner。
  • 重定位为顾问式 + 可插拔检查门禁（对外一个状态）。
  • 评审复用 PR-Agent；自治与离线学习回路。
  • 工程化收尾：栈规则、三档验证、接口命名与实现一致。

已知风险（诚实记录）

  • 验收规格来源：author 自报规格可被规避 → 已改为人核准优先（acceptance.yaml）。
  • 充分性 ≠ 意图正确：覆盖/变异证明"测得够"，不证明"测的是对的意图"——故高风险仍要人仲裁。
  • 校准依赖人审信号：人审质量不稳时，吻合度会失真；保守取并、不全凭单一信号。
  • verify 尚未大规模实跑：默认关，需在真实仓库上验证稳定性与成本。

提请评审：定位与边界是否认可，验证子系统的落地优先级，以及自动合并开启的前置条件。
