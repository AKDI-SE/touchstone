详细设计 · 数据采集（TF-GRPO 落地第一步）

Touchstone · TF-GRPO 数据采集与真值集重建

要让 TF-GRPO 真用起来还出好效果，最值钱的一份数据是「每条发现 → 人最终怎么处理了」的干净标注。本设计沿用 calibrate 既有的原则——不另建数据库，一切从 GitHub 重建：采集只做一件新事，即评审时把需要记录的信息写进评论 marker（尤其是「注入了哪些经验」）；其余结局从 GitHub API / GraphQL 线程 / git 历史重建，再装配成提供给 TF-GRPO 的真值集。

基线 v0.1.0 原则 不建库 · 从 GitHub 重建 成败关键 每条发现的人审结局 配套 learning-loop-design.html

01

设计目的

这是一次在已有系统上增加功能：给现有的 calibrate 重建管线补上 TF-GRPO 所需的数据，而不是新起一套数据基建。

先说清楚这批数据的用途这里采的是评分标准（奖励真值），不是训练数据。TF-GRPO 不训练、不更新模型权重；这批「历史 PR + 人采纳了哪些发现」只用来给评审打分，据此蒸馏出经验、加进提示词——改变的是经验库，不是模型。

现状（基线 v0.1.0）。calibrate 不建库——它从 GitHub 重建：Touchstone 的发现/风险来自评论里的 <!-- touchstone-result --> / <!-- touchstone-finding --> marker，人审裁决来自 PR 的 review 状态（APPROVED / CHANGES_REQUESTED）与是否合入，finding 级采纳来自 GraphQL 评审线程的 isResolved（线程被解决 ≈ 该条被采纳）。aggregate() 由此产出 by_rule 的 fires / changes_requested_rate / adoption_rate，已经在供计数式蒸馏与退役使用。

差距。TF-GRPO 的三个消费点要的东西，现有重建只覆盖了一部分：

消费点（learning_loop）需要的数据现状
score_review(review, human_adopted)每个历史 PR 上，人最终采纳了哪些发现（按类型/条目）仅有 PR 级 + 线程级采纳代理，未装配成「每 PR 的采纳集」
_distill_via_llm(ground_truth, …)一份真值集：PR 的 diff + 人采纳集（+ 仓/栈/混淆因素）未实现重建
graduate(store, ab_results)每个 finding_type 的 注入 vs 不注入两臂 {seen, adopted}无——marker 未记「注入了哪些经验」，无法分臂
retire / distill_candidatesby_rule 的 fires / adoption_rate已有（沿用 aggregate）

目的。定义清楚 TF-GRPO 要采什么、并给出不建库的采集方式：评审时把缺的 provenance（最关键是「注入了哪些经验」）写进 marker；其余从 GitHub / git 历史重建；再新增两个纯装配函数 build_ground_truth 与 aggregate_ab，把重建结果整理成 _distill_via_llm 与 graduate 能直接使用的结构。本设计只到「数据形状与接口」，真正的埋点与重建实现是随后一步。

02

设计逻辑与概念空间

把采集拆成「评审时留一点、离线重建一片」。评审路径只多写一条更完整的 marker（毫秒级、无额外服务）；其余靠 calibrate 周期性从 GitHub 拉取重建。下面先建概念，再给数据流。

2.1 概念

概念定义 / 由什么构成
评审记录评审时为每条发现写入 PR 评论的 touchstone-finding marker；在现有字段上扩充出 file/line、finding_type、注入的经验 id 列表、模型/端点、时间戳。它不是数据库，就是评论里的一段 JSON 注释。
采纳证据一条发现「被人接受」的来源，分可信度等级：① 显式反馈（最干净，见 2.3）；② 自动关联（线程 isResolved、被标位置随后有改动、suggestion 被 apply）；③ 推断（PR 合了但该位置没动 → 记为忽略/误报）。
漏报人审评论里指出、而 Touchstone 没提的问题——对应奖励里的 miss 项，是提升召回的关键，但需把人评论映射到代码位置（最难，分期做）。
真值集装配出来的、供 _distill_via_llm 的产物：每个历史已合 PR 的 {diff 引用, 人采纳集, （可选）漏报, 混淆因素}。
注入对照同一 finding_type 在「注入了该经验的 PR」与「没注入的 PR」两组上的采纳率——由 marker 里的注入经验 id事后切分历史即可重建，无需另存。供 graduate。
PR 级结局合入/关闭、评审轮数、合入后 N 天内是否被 revert / hotfix / 关联事故——从 git 历史与 PR 元数据重建，供 govern 退役与（可选）自治。
混淆因素reviewer 身份、作者是 agent 还是人、PR 大小、文件类型、时间。不记则会把「讨某人喜欢」误当「对」。

2.2 数据流

2.3 三个核心取舍

取舍 1 · 奖励可信度（最关键）TF-GRPO 朝奖励方向优化，奖励测得糙，就会把噪声当信号学进去。显式反馈（每条意见上的 👍/👎 或「有用/已知/误报/已改」）信噪比最高，但需要一个轻量反馈通道——在不建库的前提下，可用评论 reaction 或作者回复关键词承载，仍从 GitHub 重建。退而求其次用自动关联（线程 resolved / 位置改动），最次用推断。本设计把三档都纳入 采纳证据，每条标注来源与可信度，让奖励可以分档加权、也便于抽检。
取舍 2 · provenance 必须评审时留「这条发现产生时注入了哪些经验」只能评审当时写进 marker，事后无法从 GitHub 反推。缺了它：注入对照无法分臂（graduate 失效）、坏经验无法单条归因与回退、抽检无从谈起。所以扩充 marker 的第一要务是 injected_experience_ids（已在 orchestrator result marker 落地：learning_loop.active_ids），其次才是 file/line/模型。
取舍 3 · 类型级先行、位置级择机当前 score_review 只按 finding_type 的集合重合度算命中/噪声（见 learning-loop §3，已知简化）。marker 与采纳证据里保留 file/line/内容，就为日后把奖励从类型级升级到位置级预留了字段；先做类型级，位置级作为奖励函数的后续增强，不阻塞采集落地。

03

核心数据结构

3.1 扩充后的 touchstone-finding marker（评审记录）

在现有 marker（rule_id / agent）上补 provenance。它是写进 PR 评论的一段 JSON 注释，不是表：

字段含义新增?
finding_id该条发现的稳定 id（PR内唯一，关联采纳证据）新
rule_id / finding_type发现类型（PRA-* / 规则号）有
agent来源（pr-agent:suggestion / review）有
file · line_start · line_end定位（位置级奖励与位置改动关联用）新
injected_experience_ids[]产生时注入的 active 经验 id 列表（注入对照分臂、单条归因）新·关键
model · endpoint评审用的模型/端点（模型/端点变化时归因）新
ts时间戳新

3.2 扩充后的 CalibrationRecord（重建产物，每 PR 一条）

在现有 record（pr / risk_band / findings / finding_adoption / human_state / merged / merge_commit_sha）上补：

字段含义
finding_adoption[]每条：{finding_id, adopted(bool), evidence: explicit|thread_resolved|code_changed|inferred, fidelity}（现有仅 thread_resolved，扩成多档）
injected_experience_ids[]该 PR 评审注入过的经验 id 并集（从 marker 汇）——注入对照分臂用
reviewer · author_kind决定性 reviewer 的 login；作者是 agent 还是人（混淆因素）
size · file_types[]{files, additions, deletions}；涉及的文件类型（混淆因素）
pr_outcome{merged, review_rounds, reverted, hotfix_within_days, incident}（govern / 自治用）

3.3 真值集 GroundTruthPR（供 _distill_via_llm）

字段含义
pr_id · repo · stack定位与分类（经验按 仓·栈·发现类型 归类）
summary · diff（或 diff_ref）PR 摘要与 diff；diff_ref 可用 base..merge_commit_sha 惰性取
human_adopted[]人采纳集：被采纳发现的 finding_type（位置级时为 finding_id）。score_review 的标准答案
misses[]（可选，分期）人指出而未被覆盖的问题 {file,line,note}
confounders{reviewer, author_kind, size, ts}

3.4 ab_results（供 graduate）

形状与 graduate 现有入参一致，由 aggregate_ab 按 marker 的 injected_experience_ids 切分两臂得到：

键值
finding_type{with_seen, with_adopted, without_seen, without_adopted}——注入该类型经验的 PR 为 with 臂，未注入为 without 臂

04

接口定义

评审侧只加一个「写扩充后的 marker」的动作；离线侧给 calibrate 加两个纯装配函数，并在 main() 的重建里多拉几样 GitHub 数据。

接口（建议名）功能位置
render_finding_marker(finding, injected_experience_ids, model) -> str评审时为一条发现生成扩充后的 marker 文本（含 finding_id/file/line/注入经验/模型/ts），贴进 PR 评论。唯一的「采集写入」，写进评论、不入库review_provider / orchestrator
build_ground_truth(records) -> [GroundTruthPR]纯函数：从扩充后的 records 装配每 PR 的真值（人采纳集 + 混淆因素 + diff 引用），供 _distill_via_llmcalibrate（新）
aggregate_ab(records) -> {finding_type: arms}纯函数：按 injected_experience_ids 把历史 PR 分注入/未注入两臂，按 finding_type 汇 {seen, adopted}，供 graduatecalibrate（新）
reconstruct（main 内增强）重建时多拉：PR diff/size/author（REST）、reactions（REST，承载显式反馈）、revert/hotfix（git log / commit message）→ 填扩充后的 recordcalibrate.main（增强）
aggregate(records) -> by_rule（既有）不变；records 扩充后照常产出 fires/adoption_rate，供 distill_candidates/retirecalibrate（沿用）

整体流程。评审时 render_finding_marker 把 provenance 留进评论；calibrate.main 周期性从 GitHub 重建出扩充后的 records；build_ground_truth(records) → 真值集 → _distill_via_llm 蒸馏候选经验；aggregate_ab(records) → graduate 做 shadow A/B 门控；aggregate(records) 照旧供计数式与退役使用。三条消费链共享同一批重建出来的 records，互不阻塞。

05

一致性校验

对照每个消费点逐一核对，重点记已发现并接受的遗留项。

接口完备性score_review 要的 human_adopted ← build_ground_truth 的人采纳集；graduate 要的两臂 ← aggregate_ab（靠 marker 的注入经验 id 分臂）；retire/distill_candidates 要的 by_rule ← 既有 aggregate；单条经验归因/回退、抽检 ← marker 的 injected_experience_ids + finding 级采纳证据。各消费点均有对应来源。
概念一致性「采纳」一词全设计统一指采纳证据（带可信度等级），不与人审 PR 级 verdict 混用；finding_type 是否算评审类，与 learning_loop 的 _is_review_type 判定一致——确定性 contract 类型不进真值集的经验侧。
接受的遗留项① 漏报采集需把人自由文本评论映射到代码位置，最难——先不采 misses、TF-GRPO 先优化精确率，召回作后续增强。② 显式反馈依赖一个反馈通道（reaction/回复关键词），是个小的产品依赖；未上线前用自动关联+推断作回退，可信度等级会如实标注。③ 位置级奖励需 marker 与采纳证据都带 file/line，schema 已预留字段，score_review 的升级是后续，不阻塞采集。④ 重建依赖 GitHub marker 持续存在（评论不可被清理）——与 calibrate 现有假设相同。
量级提醒经验按 仓·栈·发现类型 分类，每一类都要积累足够的带结局样本，才能稳定蒸馏出经验（蒸馏下限 DISTILL_MIN_FIRES=8、达标每臂 ≥20）。论文整体几十条真值即可，但分到每一类需要数十个实例，意味着在活跃仓上连续采集若干周——所以从第一天就埋点，尤其是注入经验 id 与模型版本这类事后不可重建的 provenance。
