集成设计 · Integration Design

Touchstone on PR-Agent：复用开源评审引擎，叠加自有门禁与验证

既然开源的 PR-Agent 在公网可用（本工具评审公网 GitHub PR），评审与聚合这层就复用它，不再自研委员会与共识聚合器；Touchstone 只做 PR-Agent 没有的那几层——发现归一、裁决映射、单一统一门禁、verify 正确性验证、自动合并、契约来源治理、校准。一句话：把评审的轮子交给 PR-Agent，把"裁决/门禁/验证/自治"留给自己。

设计时间 2026-06（基于主文档 v3.8 / 82 提交） 定性 复用开源 + 边界重划重构 前提 互联网：公网 GitHub + 可访问的 LLM 端点
不造轮子的边界PR-Agent 的核心能力是"多工具评审 + 建议去重/排序/措辞"，正是与自研委员会重叠最深的一块——自研它是重复造轮子。真正没有开源对应、构成 Touchstone 价值的是：可插拔质量门禁（单一统一门禁）、verify 验证子系统（独立验收测试 + 充分性校验 + 变异）、可选 autonomy（自动合并 + 经验达标 + 熔断）、契约"只定位不采信"的来源治理、校准 / TF-GRPO 经验演进。本设计把工程力集中在后者，前者复用 PR-Agent。

§ 1

设计目的

本设计的定性是"复用开源 + 边界重划"的架构重构：把自研的"评审委员会 + 启发式共识聚合器"替换为开源 PR-Agent，并把 Touchstone 独有的门禁/验证/自治层叠在其上。不是新增模块，而是把一段重复造的轮子换成现成的轮子，并重新划清两边的职责边界。

1.1 现状（设计锚点）

锚点：主设计文档 v3.8 / 仓库 82 提交 / 2026-06。前提：本工具运行在互联网上（评审公网 GitHub PR、调可访问的 LLM 端点），PR-Agent 同环境可用；不涉及 air-gapped / 内网约束。当前 Touchstone 由以下部分组成：

自研评审委员会 + aggregate()
多专才产出 Finding，启发式聚合（置信过滤 + 精确去重 + 规则类定级）。← 与 PR-Agent 高度重叠的部分
contract_check
提交契约一致性核对（"只定位不采信"），产出确定性发现。PR-Agent 不做。
checks / 统一门禁
可插拔检查框架，聚合后对外发一个汇总状态检查。PR-Agent 不做。
verify
可选正确性验证子系统：独立验收测试 + 充分性校验 + 变异。PR-Agent 不做。
autonomy / calibrate / govern
可选自动合并（读统一门禁）+ 经验达标 + 熔断；校准与经验演进。PR-Agent 不做。

1.2 期望与纠偏

期望：用 PR-Agent 当评审引擎，退役自研评审专才与共识聚合器；独有层不变地叠在其上。需要在此澄清两点用户期望之外、但必须面对的事实：

输出模型不一致PR-Agent 产出的是 review/suggestion 文本项，与本系统的 Finding（file/line/rule_id/category/severity/confidence）模型不同。必须有一层"发现归一"把前者映射成后者，否则风险路由、统一门禁、校准都无从对接。
PR-Agent 不产出裁决它不发单一统一门禁、不做风险路由（三类判定 / 风险等级 / 验证预算分配）、不做正确性验证、不做契约一致性核对。这些是 Touchstone 的职责，必须保留，不能因为"复用 PR-Agent"而误以为它们也被覆盖了。
不可并存两套评审若既保留自研委员会、又接入 PR-Agent 做同样的评审，就是"双轮子"——既增加噪声又增加维护。本设计明确：评审/聚合由 PR-Agent 取代，自研评审专才与 heuristic 共识退役。
§ 2

设计逻辑与概念空间

围绕三个核心取舍说明设计选择，并定义为描述这套集成而引入的新概念，最后给出两个逻辑视图（职责分层、数据流）。

2.1 概念空间

评审提供器 Review Provider
把 PR-Agent 抽象成一个可替换的"评审观察来源"：输入一个 PR，输出一批原始评审条目。封装了"怎么调 PR-Agent（CLI/API、公网或自托管端点、注入指令）"，对上层只暴露"给我这个 PR 的评审观察"。
发现归一 Finding Normalization
把 PR-Agent 的 review/suggestion 项映射到本系统的 Finding。封装了两套模型之间的字段对齐与默认值（严重度、置信度、category 的推断）。
裁决映射 Verdict Mapping
把归一后的 Finding 映射到三类判定 + 风险等级 + verification_decision（风险路由所需）。它不做共识（共识已由 PR-Agent 完成），只做分类与定级——是原 aggregate() "评审侧"职责的瘦身版。
退役边界 Retirement Boundary
明确划线：哪些自研代码因 PR-Agent 接入而退役（评审专才 prompt、heuristic 共识），哪些保留（contract_check、checks、verify、autonomy、calibrate、govern）。封装了"复用到哪、自研留哪"的决策，防止双轮子或误删独有层。

2.2 核心取舍

取舍一：边界划在哪——PR-Agent 管"观察"，我们管"裁决"

PR-Agent 负责评审观察与建议（广度、措辞、多语言，已成熟）；Touchstone 负责裁决与门禁（风险路由、统一门禁、验证、自治）。封装策略取"薄封装"：把 PR-Agent 当评审提供器，用它的注入点（extra_instructions / best-practices）传入我们的规范，而不去 fork、改它的源码——否则要长期承担与上游分叉的维护负担，背离"不造轮子"的初衷。

取舍二：聚合归谁——不自研共识，只做裁决映射

PR-Agent 自带建议去重、排序、按重要性过滤（其核心竞争力）。所以不自研带共识算法的聚合器。但风险路由与统一门禁需要"三类判定 + 风险等级"，这是 PR-Agent 不产出的，因此保留一个轻量裁决映射：把归一后的 Finding 按 category/严重度映射到风险等级与验证预算决策。它取代 aggregate() 中"评审侧"的职责，但去掉了"共识"那部分（交给 PR-Agent）。

取舍三：怎么不丢掉独有价值——明确 PR-Agent 不覆盖项

逐项确认 PR-Agent 不覆盖、因而必须由我们保留的能力：单一统一门禁（汇总状态检查）、风险路由、verify 正确性验证、契约一致性核对（contract_check）与来源治理（human_curated）、校准 / 经验演进。其中 contract_check 尤其关键：PR-Agent 不做"提交契约 vs 改动"的确定性核对，它继续作为 touchstone-rules 的输入输入到统一门禁。

§ 3

核心数据结构

只定义与设计逻辑对应的关键成员；复用主文档已有的 Finding / RiskAssessment / CheckResult / VerificationResult，不重复定义。

结构关键成员说明
ReviewItemkind(review/suggestion) · file · line_range · body · label? · tool?PR-Agent 单条输出的抽象。评审提供器把 PR-Agent 的原生响应解析成一组 ReviewItem，屏蔽其具体格式。
Finding（复用）file · line · rule_id · category · severity · confidence · agent归一目标。agent 记来源（如 pr-agent:review / pr-agent:suggestion / contract-check），供校准按来源归因。
NormalizationMaplabel→category · 默认 severity · 默认 confidence · 丢弃规则归一策略表，写在配置里（如 .touchstone/pr-agent.yaml）。把 PR-Agent 的工具/标签映射到我们的 category 与默认严重度/置信度；含 high_categories（哪些 category 升 high，默认 [security, correctness]），定级策略显式可调而不改代码。
RiskAssessment（复用）risk_band · blast_radius · verification_decision裁决映射的产出，结构与主文档一致，下游风险路由/统一门禁/autonomy 直接复用。
§ 4

接口定义

以函数注释形式给出新增接口；复用的接口（checks / verify / autonomy）签名不变，只说明衔接关系。

# 新增：评审提供器（封装 PR-Agent 调用，可替换）
review_provider.fetch(pr_ctx) -> list[ReviewItem]
    # 对 pr_ctx（owner/repo/number/token）经子进程适配器 pr_agent_runner 调 PR-Agent，
    # 传入我们的规范作为 extra_instructions/best-practices，解析其响应为 ReviewItem 列表。
    # 屏蔽 PR-Agent 的具体调用方式与输出格式。

# 新增：发现归一
normalize(items: list[ReviewItem], nmap: NormalizationMap) -> list[Finding]
    # 按 nmap 把每个 ReviewItem 映射成 Finding：推断 category、给默认 severity/confidence、
    # 标 agent 来源；按丢弃规则过滤明显无关项。

# 新增：裁决映射（取代 aggregate 的"评审侧"，不做共识）
map_verdict(findings: list[Finding], rule_index) -> tuple[list[Finding], RiskAssessment]
    # 把归一后的 Finding 按 category/severity 映射到三类判定与风险等级，
    # 产出 verification_decision（风险路由）。复用主文档 aggregate 的"定级"骨架，去掉"共识/去重"。

# 复用（签名不变）：
contract_check.check_contract_consistency(diff, contract, rule_index) -> list[Finding]   # 契约确定性核对，PR-Agent 不做
checks.run_checks(cfg, pr) / checks.post_gate(pr, cfg, results)                           # 聚合并发单一统一门禁
verify/verify_change.py                                                                   # （可选）正确性验证，折进统一门禁
autonomy.decide_auto_merge(risk, findings, loop_decision, gate, ...)                      # （可选）读统一门禁自动合并
关系总结一个 PR 来：review_provider.fetch 从 PR-Agent 取评审观察 → normalize 成 Finding → map_verdict 出风险等级；与此并行，contract_check 做契约确定性核对（PR-Agent 不做）。两路 Finding 一起作为 touchstone-rules 输入，经 checks.run_checks + post_gate 发那一个统一门禁；高风险且开启时（可选）verify 折进统一门禁，autonomy（可选）读统一门禁替人点合并。默认形态仍是人点合并。
实现状态（v0.2）已落地离线骨架：touchstone/review_provider.py（parse_pr_agent / normalize / map_verdict / load_nmap / PRAgentProvider）+ .touchstone/pr-agent.yaml + 12 个离线测试（解析/归一/裁决映射/注入与集成点），全绿。PR-Agent 调用已实现为子进程集成（_invoke_endpoint 起 pr_agent_runner 子进程，v0.7）；离线经 pr_agent_output 注入测 plumbing。已切换（v0.4）：orchestrator.py 默认走 PR-Agent 路径，自研委员会（专才编排 + 共识聚合器 + 相关 LLM 调用）已退役删除；run.py 与测试同步更新。PR-Agent 是 Apache-2.0 的 pip 包、不是要部署的服务：集成 = 子进程里 pip install pr-agent 并调它。端到端真跑只差凭据：子进程环境的 LLM key（pr-agent 经 LiteLLM 调）+ GitHub token——这是任何 AI 评审器固有的，非 PR-Agent 特有，也无需"部署 PR-Agent"。无 shadow A/B 门槛——采用 PR-Agent 的依据是"不造轮子/不维护自研评审引擎"，而非"评分更高"；若评审质量需提升则调 PR-Agent（extra_instructions/best_practices/换模型），不回退委员会。
LLM 接入与反静默故障（生产化）线上 touchstone.yml 已含一步把 pr-agent 装进独立 venv（.pragent-venv），并用 TOUCHSTONE_PRAGENT_CMD 指过去——遵循"PR-Agent 不进本仓依赖"原则。LLM 凭据经仓库 secrets LLM_BASE_URL/LLM_API_KEY/LLM_MODEL 提供，pr_agent_runner.run 把它们映射成 PR-Agent/LiteLLM 认的键（OPENAI_API_KEY/OPENAI_API_BASE + config.model = openai/<LLM_MODEL>），集中在一处、不散落到 workflow 明文 env。关键：防静默故障——评审引擎层任何失败（pr-agent 没装 / LLM 端点 401/超时/不可达）都不会被吞成"0 条发现"：pr_agent_runner 不抛、不非零退出，而是返回结构化 {"_degraded": "no_engine"|"llm_failed", "reason": ...}；review_provider._invoke_endpoint 转成 ReviewEngineDegraded 异常；orchestrator.review_pr 捕获后在返回里带 engine_status，post_results 经 _engine_banner 把"⚠️ AI 评审未运行 / LLM 调用失败，本次只含确定性核对"写在贴到 PR 的评审评论顶部、并反映在 check 标题里。降级时仍跑确定性契约与栈规则核对（它们不依赖 LLM），只是把"AI 评审没生效"明白告诉人。本地 run.py 同样打印该说明。
规范注入我们的单一事实源规范（.touchstone/standards.yaml）中"可机器自动核对"的条目继续走确定性 touchstone-rules；"需主观判断"的条目作为 extra_instructions/best-practices 注入 PR-Agent——规范一处定义，分别提供给 PR-Agent（查主观问题）与确定性检查（查可核对问题）。已落地（v0.6）：touchstone/gen_best_practices.py 从 standards.yaml 取 machine_checkable=false 的规则，按 PR-Agent 推荐的 pattern-based 结构生成 .touchstone/best_practices.md（违反即标 “Organization best practice”，归一为 convention）；可机检规则不进该文件、继续走确定性检查。部署时把该文件放到 PR-Agent 读取处（仓库根，或经 PR-Agent 的 metadata.yaml 路径映射）；改 standards.yaml 后重新生成（测试含"生成结果须与提交版一致"的防漂移校验）。
生产接通踩坑（实证记录）把 pr-agent 从"离线单测可跑"接到 GitHub Actions 真跑，逐层踩了四个坑，皆因离线测不到、且 pull_request_target 取 base 分支的 workflow+代码、PR 改动无法在本 PR check 自证：
① Python 3.14 编译失败——pr-agent 的 tiktoken 经 PyO3 ≤3.13，setup-python "3.x" 解析到 3.14 即源码编译失败；钉 3.13 + 安装步 continue-on-error（装失败让 orchestrator 降级而非整 job 崩）。
② 取 PR 无凭据——pr-agent 需 settings.github.user_token，runner 只配了 LLM 侧；从 GITHUB_TOKEN 注入 + 显式 git_provider="github"。
③ stdout 污染——LiteLLM 把 LiteLLM.Info 等 print 到 stdout，与 runner 的 JSON 混淆致 json.loads 失败、误判 no_engine；运行期 fd 级把 stdout 重定向到 stderr。
④ 降级分类——取 PR 失败（pre-LLM）与 LLM 失败要分 provider_failed/llm_failed，否则横幅误导。教训已写进 CLAUDE.md DoD「CI 烟测」：动 workflow/依赖/环境要验在真 runner 跑通，且 pull_request_target 改动合入后须用下一个 PR 回归。
§ 5

一致性校验

对设计目的、逻辑、数据结构、接口做综合校验；下面记录结论，重点写发现并已处理的问题与有意接受的遗留。

概念一致Finding / RiskAssessment / 统一门禁 / 风险等级 与主文档同名同义；新概念（评审提供器 / 发现归一 / 裁决映射 / 退役边界）全文一致使用。
层次一致PR-Agent 只做评审观察，不发统一门禁、不做风险路由、不做验证、不做契约核对——没有把本系统的裁决/门禁职责下推给 PR-Agent；本系统也不再自研评审/共识——没有在我们层重复 PR-Agent 的评审。边界干净。
接口完备评审获取（fetch）、归一（normalize）、裁决映射（map_verdict）、契约核对、发统一门禁、可选验证、可选自治，每个操作都有对应接口。
退役与去重（防双轮子）因 PR-Agent 接入而退役：自研评审专才 prompt、aggregate() 的共识/去重部分。保留：contract_check（契约一致性，PR-Agent 不做）、checks/统一门禁、verify、autonomy、calibrate/govern。明确不并存两套评审。
校准维度调整校准（calibrate）的归因维度由"按专才"改为"按 PR-Agent 工具/规则 + contract_check"——Finding.agent 记来源即可支撑；退役噪声来源的机制不变（只是退役对象从"某专才"变为"PR-Agent 某类建议"）。
有意接受的遗留① PR-Agent 的评审质量依赖其 prompt 与模型，本设计通过 NormalizationMap 的默认严重度/置信度 + 校准持续治理噪声，但质量是否优于自研委员会，需在真实 GitHub PR 上跑 shadow A/B 验证（本沙箱无实时 GitHub/LLM 凭据，故在你的环境做）。② review_provider 对 PR-Agent 调用方式（CLI vs API）的最终选型，待真实环境联调确定；本设计只约束"评审提供器"这一抽象边界，不锁定调用细节。
