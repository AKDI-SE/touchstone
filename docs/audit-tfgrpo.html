01

总评

问题一 · 方法命名夸大没用 GRPO 的定义性机制（组内相对数值优势），实质是 group-sampling + LLM critique-distillation，叫 TF-GRPO 是营销性拔高。
问题二 · 「自进化」默认惰性达标门控（graduate 的 A/B）与数据管线（ground_truth、aggregate_ab）多半未接通 → candidate 永不自动 active，唯一进注入的是人手写种子。
问题三 · 实现与设计不符经验 id 不按仓·栈归类、多轮不迭代精化、跨 kind 冲突无消解、退化组无防护。

02

设计层

D1 · 名不副实：这不是 GRPO（最尖锐）

设计 §3.1 自己写明：论文的「数值优势 (r_i−mean)/std」→ 不用，换成「语义优势」（让旗舰模型内省高分 vs 低分评审）。

但 GRPO 的「R」(Relative) 指的就是组内相对数值优势归一化——这是它降噪、降方差的定义性机制。本实现只保留了 GRPO 的词汇（rollout / group / advantage），把核心数学换成了「LLM 看两份评审总结一句话」。这在算法分类上是 group-sampling + LLM critique-distillation，不是 GRPO。代价实打实：

  • distill_semantic_advantage（learning_loop.py:184）只取 ranked[:2] 和 ranked[-2:] 做对比。G=4 时有效样本就 2 高 vs 2 低，方差极高——一份评审碰巧命中了人关心的类型就「高分」，并不代表它代表可泛化策略。GRPO 的全组归一化正是用来压这个方差的，这里被丢了，设计未承认这一成本。
无论 arXiv 2510.08191 具体如何（无法核实其精确内容），本实现独立地看就是「冻结模型 rollout + LLM 内省蒸馏」。建议改名或显式声明「只取 group-sampling 思想、不用其优势归一化」。

D2 · 「自进化」默认是惰性的（最实际）

设计画的闭环：distill → candidate → shadow A/B 达标 → active → 注入。代码实际：

环节实现?
distill_* 产 candidate✅
merge_candidates 入候选池✅
candidate→active 自动通道（graduate 的 ab_results）❌ 无生产者
ground_truth（TF-GRPO 输入）❌ 无 build_ground_truth，只能手填 JSON
injected_experience_ids（A/B 分臂依据）⚠️ marker 已埋 injected_types，但消费端 aggregate_ab 不存在

grep 确认：build_ground_truth / aggregate_ab 全仓不存在；main() 里 graduate 只在 TOUCHSTONE_AB_RESULTS 文件存在时才跑，而那份数据没人产。

净效果TF-GRPO 蒸出的 candidate 永远不会自动变 active；唯一能让经验进注入的是人手写 seed_experience。所以「评审越用越准」的自动闭环不通——它是个「人写种子 + 一个产出永不激活的候选的离线脚本」。这是「自进化」卖点和实现落差最大的地方。

D3 · 奖励粒度过粗

score_review（learning_loop.py:125）把评审和人采纳都当成 finding_type 集合做交并差。但 finding_type 是 PRA-SECURITY / PRA-POSSIBLE_BUG 这种粗类型——两份评审在不同文件报了同一个 PRA-SECURITY，都算同一个 hit。hits/noise 因此都很粗，奖励信号噪声大。设计 §3.3 没讨论这个粒度问题。

D4 · 奖励有偏（人审盲区）—— 已承认、应对合理

优点坑 2 正是此点；应对（advisory-only + protected types + locked + contract_check 作固定基准）是想清楚了的好设计。

03

实现 vs 设计 的具体差异

I1 · 经验 id 不含 repo/stack（与设计声称矛盾）

设计 §3.3 / §3.5 反复说「严格按 仓·栈·发现类型 归类」，Experience 也确实有 repo / stack 字段——但去重键 _exp_id（learning_loop.py:65）只是 f"{kind}:{finding_type}"，不含 repo/stack。多仓部署下，A 仓的 emphasize:PRA-X 会和 B 仓的撞成同一条、互相覆盖。要么 id 纳入 repo/stack，要么文档明确「当前仅单仓」。

I2 · 多轮 epoch 不更新 conditioning E（与「逐轮修正」矛盾）

_distill_via_llm（learning_loop.py:253）在循环外算一次 experience_text = render_injection(store)，然后所有 epoch、所有 PR 的 rollout 都用这同一份 E 做条件。设计 §3.1/§3.2 说「经验库逐轮累积、修正」「下轮注入」，但代码里 distilled candidate 进的是局部 acc，不回灌进 conditioning E。所以 epochs>1 只是多攒几条 candidate（且因 LLM 随机可能出变体），不是「迭代精化 E」。要么每轮用上一轮结果重渲染 E，要么文档别说「逐轮修正」。

I3 · 同类型的 emphasize/suppress 可并存且都被注入

emphasize:PRA-X 与 suppress:PRA-X 是不同 id → merge_candidates 各存各的；一旦都 active，render_injection 会把「该挑 PRA-X」和「别挑 PRA-X」同时塞进 PR-Agent 的 extra_instructions —— 矛盾指令。无跨 kind 冲突消解。设计 §6 只提醒了「同 kind+type 别写两条」，没处理跨 kind 冲突。

I4 · 退化组无防护

distill_semantic_advantage 取 top-2/bottom-2；若整组 reward 同分（如都是空评审、或 LLM 返回全 []），high 与 low 重叠/无差异，内省「对比」无意义，仍会强制产出（多半是垃圾）经验。无「组内方差为 0 / 有效评审过少则跳过」的守卫。

04

测试 / 验证

  • 全程 _fake_llm（tests/test_learning_loop.py:127 等确定性假 llm）：只验证 plumbing——步骤串联、JSON 解析、protected/locked 生效、score_review 精确值。蒸馏的「质量」、以及「注入后采纳率是否真升」从未被验证（离线也确实测不了 LLM 质量）。这点设计 §3.5 是诚实的（「离线以假 llm 覆盖测试」），但要清楚：TF-GRPO 的核心价值（让评审更准）目前是未经验证的假设，且 D2 说连验证它的 A/B 管线都没接。

05

值得肯定的地方

架构评审 / 学习解耦：PR 关键路径只读经验库、毫秒级；学习挂了评审用上一版。
安全模型advisory-only + 经验不进合入闸 + protected types + locked + 可单条 disable + contract_check 作固定基准——防「回路放大人审盲区」的几道闸都想到了。
可插拔register_distiller + rollout/score/distill 三步可注入，扩展性好。
奖励奖励函数合理，权重可配（TOUCHSTONE_W_NOISE / TOUCHSTONE_W_MISS）。

06

建议（按性价比）

#对应建议
1D1正名：文档把「TF-GRPO」降调为「rollout + LLM 经验蒸馏」，或显式声明「只借 group-sampling，不用 GRPO 的优势归一化」。
2D2接通自进化（最该做）：补 build_ground_truth（从 calibrate 重建）+ aggregate_ab（按 marker 的 injected_types 切 A/B 两臂）→ 喂给 graduate。否则「自进化」始终是手动。
3I1id 纳入 repo/stack，或文档限定单仓。
4I2真迭代：每轮用上轮 active/candidate 重渲染 experience_text 再 rollout。
5I3冲突消解：同 finding_type 的 emphasize/suppress 不可同时 active。
6I4退化组守卫：组内 reward 方差为 0 或有效评审 < 阈值时跳过 distill。

07

附：可举证项（用真实函数可跑出）

  • 多仓 id 撞（I1）：两个不同 repo 蒸出同 kind:finding_type → merge_candidates 互相覆盖（后写覆盖先写 repo/stack/text）。
  • candidate 永不自动 active（D2）：graduate 在 main() 中仅当 TOUCHSTONE_AB_RESULTS 存在才调用；无 aggregate_ab → 候选恒为 candidate，render_injection 不注入。
  • 矛盾共存（I3）：同 finding_type 的 emphasize + suppress 都 active → render_injection 同时输出两条相反指令。
