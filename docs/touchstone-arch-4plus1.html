§ 0 · 概述

系统是什么

Touchstone 是插在「PR 提交」与「人决定合入」之间的一层自动评审：默认只给高信噪比的评审意见，合入按钮始终由人来点。评审本身复用开源 PR-Agent；Touchstone 自己只做 PR-Agent 没有的部分——把评审输出归一成本系统的 Finding、按风险裁决映射、做确定性的契约与栈规则核对、可选的正确性验证（verify）、可选的自动合并（autonomy），以及离线的校准与学习回路。

贯穿全系统的一条红线：评审只产建议与风险分流，不产准入；真正"必须通过才能合"的硬质量门禁由可插拔检查聚合成对外一个状态，由人或（可选的）自动合并据其放行。

§ 1 · 逻辑视图　Logical View

职责分层与功能模块

关注"系统由哪些职责单元组成、它们之间的关系"。Touchstone 按决策权分四层：人持有治理与最终裁决；判断层只产建议与风险；质量门禁层做客观、可复现的准入核对；离线层用真实结果反过来校准前面几层。

图 1 · 四层职责：向下守底线（客观验证），向上提上限（AI 判断），顶层人治理，离线层用真实结果回灌校准。判断层全错也越不过门禁。

§ 2 · 进程视图　Process View

运行期的并发与编排

关注"运行时有哪些进程/任务、如何并发与协作"。Touchstone 不是单进程，而是 GitHub Actions 上分职责的多个 job，外加几条离线 cron。职责分离是设计原则：判断层与客观验证不在同一进程里跑。

图 2 · 多 job 编排 + 离线 cron。两层只经"经验库 / 达标类"耦合：离线学习挂了，同步评审仍用上一版经验照常跑。

并发与边界

  • 职责分离：touchstone（判断）/ verify（客观验证）/ gate（聚合）/ auto_merge（放行）分属不同 job，判断层不在同一进程里做验证。
  • 对外只一个状态：touchstone 的 check 永远 neutral 不拦；是否硬拦只由 gate 发的 touchstone/gate 决定。
  • 校准串行：calibrate 经 GitHub API 重建、加退避，不上并发（GitHub 二级限流对并发有惩罚）。
  • 同步/离线解耦：同步链路毫秒级、无学习开销；学习只在离线 cron 发生。

依赖与复用边界

  • 复用（不进仓库）：pr-agent（Apache-2.0，装在独立 venv，经子进程适配器 pr_agent_runner.py 调用）——评审广度全靠它。
  • 成熟库依赖：unidiff（diff 解析）、requests（GitHub I/O）、openai（LLM 调用，支持自定义 base_url）、pyyaml。
  • 自写仅限本系统独有逻辑：归一/裁决映射、契约与栈规则、风险分流、verify 子系统、自治/治理/校准/学习。
  • 零跨模块隐式耦合：同步层与离线层只经经验库耦合。
§ 4 · 物理视图　Physical View

部署与运行环境

关注"软件跑在什么硬件/服务上、如何连通"。Touchstone 运行在互联网上：评审公网 GitHub PR、调一个可访问的 LLM 端点；不依赖任何内网/专有后端服务。

图 3 · 部署拓扑。CI Runner 上跑 touchstone 各 job + 独立 venv 的 pr-agent；对外只连 GitHub 与一个 LLM 端点；不依赖任何外部存储服务。

运行环境约束

  • 互联网，不是内网：评审公网 GitHub PR、调可访问的 LLM；PR-Agent 是 pip 包，集成 = 依赖 + 子进程调用，不是部署端点。
  • 凭据：GitHub token 与 LLM key 走环境变量；可经代理访问公网。
  • 零专有后端：当前部署不依赖任何外部存储或数据库服务，对外只连 GitHub 与一个 LLM 端点。
  • 可在自有环境独立运行：python -m touchstone.run --repo O/R --pr N 可对 Actions 之外、甚至不属于自己的外部 PR 做端到端评审（默认 dry-run）。
§ +1 · 场景视图　Scenarios

用关键场景串起前四视图

4+1 中的"+1"：用几条有代表性的用例，验证前四个视图能协同支撑真实工作流。

场景 A · 普通 PR 的顾问式评审（默认形态）

  1. PR 打开 → touchstone job 取 diff，review_pr 跑 PR-Agent 评审归一 + 契约核对 + 栈规则 → map_verdict 出发现与风险等级。
  2. route 判低/中风险 → 人不必逐行看；touchstone 发 neutral check + 行内/总结评论。
  3. gate job 聚合检查 → 发 touchstone/gate；人看一眼评论与统一门禁，点合并。

场景 B · 高风险 PR 触发正确性验证

  1. 评审命中 security / 契约影响 → 风险 high、影响面严重 → route 给 full_suite。
  2. verify job（已开启时）调 verify_change：用人核准的验收规格生成独立验收测试，跑充分性校验（覆盖 → 空实现哨兵 → 变异）。
  3. gate 把 verify 结论折进统一门禁；只有"可信的通过"（人核准规格 + 验证达标）才计入。否则统一门禁非绿、回落到人。

场景 C · 机器自改的反馈循环

  1. touchstone 给出可自改的结构层发现（带 suggested_fix）→ loop_step 判 continue。
  2. 写代码的 agent 经 API 读 findings、改完 push → synchronize 再触发评审。
  3. 无可自改发现且 CI 非红 → converged；超轮次/震荡/假修 → escalate 交人。正确性始终由 verify 正交把关。

场景 D · 校准与（可选）自动合并的长期演进

  1. 离线 calibrate 从已合 PR 重建：touchstone 发现 vs 人审裁决 → 算吻合度、标噪声来源。
  2. govern 把复发且证实有害的发现固化为门禁；按变更分类影子观察、达标后发布"可自动放行"的类。
  3. 开启 autonomy 后，对已达标的类、且统一门禁绿 + 七项条件全过，才自动合并；合入后 revert/hotfix 越阈 → 熔断收回自主权。
