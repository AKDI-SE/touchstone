覆盖已采纳的评审意见 1、2、3、4、5、6、7、10（意见 8、9、11 明确不采纳，不在本修订范围）。
状态锚点：仓库 AKDI-SE/touchstone @ c34f9f7（main，2026-07 初），主设计文档 docs/touchstone-design.html 现行版。本文是对主文档与代码的修订方案，评审通过后合入主文档并按落地清单实施。

1. 修订目的

本次修订属于对已有设计的增强与纠偏，不改变系统定位（advisory-only，建议不拦截合入，准入只由确定性质量门禁产生），不推翻已有的评审主链（复用 PR-Agent）、验证子系统与学习回路。八条意见按性质归为四组：

组意见一句话目标现状差距
反馈质量1、2每条反馈给出「改到什么状态算过关」与「往哪个方向改、为什么」，不给动手级指令缺位/冲突Finding 无达成判据；suggested_fix 现定义为「可直接 apply 的补丁或精确指令」，与意见 2 直接冲突，且 loop.author_actionable 依赖该字段
收敛机制3、10双 agent 交互围绕一份可勾选的清单逐项销项；轮次记账跟内容走，重开 PR 刷不出新轮次缺位循环状态是隐藏 marker 里的发现签名集，无逐项销项载体；轮次以 PR 号记账，关旧开新即清零
文档工程4、5评审报告版面模板抽为一等设计资产；每个数据结构在设计图上有明确锚点散落版面内联在 orchestrator.render_summary；主文档 §2 视图与 §3 结构之间无对应关系
确定性事实与验收7、6修改范围由确定性工具产出为一等结构；全流程中间状态做成带真实数据的可视化页面用于验收半落地contract_check 已有 unidiff 确定性核对（SCOPE/TEST/DUP-001），但产物内部消化未成结构，blast_radius 仍由模型来源的发现类别推导；可视化交付物不在仓库

2. 修订逻辑与概念空间

本次修订引入四个新概念，它们互相咬合、共同回答「反馈怎么写、循环怎么收、账怎么记、事实从哪来」四个问题：

范围事实（ScopeFacts） —— 由确定性工具直接从 diff 计算出的修改范围事实：改了哪些文件、每个文件增删多少、是否命中契约/安全敏感路径。它不依赖任何声明、不经过任何模型，是继「硬门禁结果」「经验知识库」之后的第三个客观锚。契约一致性核对从「模型比对声明与 diff」升级为「声明集合与范围事实集合做确定性求差」。（落实意见 7）

达成判据（done criteria） —— 每条评审发现附带的、可核验的「改到什么状态算过关」描述。优先给确定性判据（对应规则复检通过、指定路径不再命中、指定测试通过），给不出确定性判据时给模型复核判据（附复核问题）。有了它，author 知道终点在哪，销项有依据。（落实意见 1）

收敛清单（ConvergenceChecklist） —— 一份写入文件并同步为置顶 PR 评论的逐项销项清单。评审方每条发现即一条清单项（含改进方向、依据、达成判据）；author 改完逐项勾选或写明不改理由；评审方按达成判据复核销项。清单销项率取代「无新发现」成为收敛指标。（落实意见 3，并为意见 1 的判据提供载体）

轮次台账（RoundLedger） —— 轮次记账主体从 PR 号改为内容指纹（由范围事实派生）。新 PR 与近期已关闭 PR 内容同源时，继承其轮次余额与未销项清单，重提刷不出新额度；人可用 label 显式授权重置。（落实意见 10）

四个概念的依赖关系：范围事实是地基（意见 7），向上支撑指纹（意见 10）与契约核对；达成判据（意见 1）写在清单项里，清单（意见 3）是循环收敛的账本；报告版面模板（意见 4）规定这些内容在评审报告中的固定位置；锚定矩阵与可视化（意见 5、6）保证以上全部结构在图上、在页面上都能被逐项核对。

3. 逐条意见的设计修改

意见 1 · 反馈必须给出可核验的达成判据

现状：Finding 只有 rationale（问题依据），无「改到什么状态算过关」；loop_step 的收敛定义是「无更多可自改发现」，是隐式终点，author 无法预判评审方下一轮是否满意。已有的 .touchstone/acceptance.yaml（human_curated）是 PR 级验收规格，用于驱动验证子系统，不解决发现级的销项判据问题——两者层次不同，命名与用途保持区分。

修改：Finding 新增 done_criteria 字段，第一轮评审即必须产出。判据分两档，确定性优先：

  • 确定性判据（评审方可机器复核）：对应规则复检通过（如 SEC-001 不再命中）、指定路径的 diff 不再包含某模式、声明与范围事实求差为空、指定测试存在且通过。
  • 复核判据（给不出确定性判据时）：一句可回答的复核问题（如「新增的回滚路径是否覆盖了跨模块调用失败的分支？」），下一轮由评审模型针对该问题定向复核，而不是重新全量评审。

收敛定义随之修改：converged = 收敛清单全部销项（done 或 waived）且本轮无新增发现。判据写在清单项里，销项即核销判据，防止「author 声称改好、评审方凭感觉再挑」的发散，也防止双方协商降级。

意见 2 · 给方向与依据，不给动手级指令

现状冲突点：主文档 §3.2 将 suggested_fix 定义为「可直接 apply 的补丁或精确指令」，且 loop.author_actionable 以「有 suggested_fix」为进入自改循环的门槛。评审方上下文不完整（看不到原始需求与关键架构文档），补丁级建议命中率低且剥夺 author 结合自身上下文决策的空间。

修改：按来源拆分建议字段——确定性来源与模型来源待遇不同：

  • fix_direction（必填）：改进方向的一句话描述，不含具体代码改法。例：「考虑将配置解析独立封装」而非「把 42–58 行剪切到 config.py」。
  • fix_reasoning（必填）：提出该方向的原因与目的，让 author 知道建议在优化什么。例：「配置模块可独立封装、独立测试，降低本 PR 的评审面」。
  • deterministic_patch（可选，仅确定性来源）：contract_check／stack_rules 等 confidence=1.0 的规则命中（如硬编码密钥所在行删除）允许给出精确修复——规则命中没有上下文歧义，不受意见 2 约束。模型来源的发现禁止填写此字段；PR-Agent 输出中的 suggestion 代码块在 normalize 阶段降级为方向描述。

author_actionable 的门槛从「有 suggested_fix」改为「有 fix_direction」；原 suggested_fix 字段废弃，报告版面第④段的每条发现按「定位 · 方向 · 依据 · 达成判据」四要素呈现。

意见 3 · 双 agent 交互改为围绕落地清单逐项销项

现状：循环状态存于 PR 评论隐藏 marker（发现签名集 history），只能做集合比较判震荡，没有逐项销项载体；人和 author 都看不到「哪几条闭环了、哪几条为什么不改」。

修改：引入收敛清单，双载体同步：

  • 置顶评论（人机共读）：机器人维护单条置顶评论，正文用 GitHub task list（- [ ]）呈现每条清单项（方向＋依据＋达成判据），评论尾部带隐藏 JSON marker 作机器可读的权威状态。task list 的勾选只是输入信号，权威状态以评审方按达成判据复核后写入 marker 的为准——沿用现有 trusted_bodies 只信机器人评论的防篡改机制。
  • 写入文件（离线审计）：每轮清单快照写入 run artifact（checklist-round-N.json），供意见 6 的可视化页面与校准回放使用。

条目状态机：open → done（author 勾选＋评审方按判据复核通过）／→ waived（author 写明不改理由，需人核准）／→ split（拆出独立 PR/issue，附链接）。非法转移（如无复核直接 done）一律回退 open 并记入报告。

震荡与无推进检测升级：从「签名集重复」改为「清单状态无推进」（连续两轮销项率为零且无 waived/split 申请），语义更准，也覆盖了「author 只发布评论不实际修改」的假修情形。

意见 4 · 评审报告版面模板抽为一等设计资产

现状：版面内联在 orchestrator.render_summary（约 §回贴段 60–95 行），设计文档里找不到版面定义，评审设计时无法挑战「什么内容出现在哪一块」。

修改：版面固定为七段，抽出为 touchstone/templates/review_report.md（带占位符的版面文件），代码只做填充；版面文件全文纳入主设计文档新章节「评审报告版面模板」，随文档一起评审、随变更历史演进：

段内容读者
① 声明与风险横幅ADVISORY 声明＋风险等级＋建议动作＋验证档人（第一眼）
② 一句话总结本 PR 在做什么、评审方总体判断人
③ 确定性事实区范围事实摘要（文件数/增删行/敏感路径命中）＋门禁状态＋同源提示（如有）人＋机
④ 逐条发现定位 · 方向 · 依据 · 达成判据（按置信降序，显示截断）人＋author agent
⑤ 收敛清单task list＋销项率＋轮次余额人＋author agent
⑥ 验证结果验证子系统输出（如触发）人
⑦ 机器状态隐藏 JSON marker（LoopState＋清单权威状态＋台账引用）机

意见 5 · 数据结构与设计图严格对应

现状：主文档 §2 五个视图与 §3 八个结构之间无对应关系；内生控制变量（MAX_ROUNDS、conf_min、_SEVERE_BLAST 等）散在代码常量里，图上无锚点，读者只能猜测某个参数是写进评审回复还是后台控制变量。

修改：主文档新增「数据结构-流程锚定矩阵」章节（本文 §6 给出全量矩阵，直接搬入）；§2.4 视图二的每个流程节点补注「使用的结构（读/写）＋代码路径」；内生控制变量单列一节，逐个标注它控制哪个环节、默认值、修改途径。此后新增任何结构或控制变量，矩阵行与图上锚点必须同步提交，作为文档合入的检查项。

意见 6 · 全流程中间状态可视化（带真实数据）

现状：可视化交付物不在仓库。设计逻辑、路径与中间状态无法用真实数据逐项核对，难以发现「看似正确、实则不符」的实现。

修改：新增交付物 docs/touchstone-visualization.html——自包含离线 HTML＋内联 SVG＋原生 JS，无外部依赖。以一个真实 PR 案例（建议用自证运行产生的真实数据，如 PR #31–#34 系列）贯穿全流程，每个节点可展开查看该节点的真实中间值：范围事实 JSON、逐条 Finding、清单各轮状态快照、LoopState marker 原文、RiskAssessment、校准记录；内生控制变量集中一栏展示当次取值。

验收标准（写进文档）：流程图上任一环节的中间值在页面上不可见，即判定该处设计或实现存在缺陷——可视化不是演示品，是验收工具。数据来源即意见 3 的 checklist-round-N.json 与既有 run artifact，页面加载本地 JSON，不需网络。

意见 7 · 修改范围由确定性工具产出为一等结构

现状（半落地）：contract_check.parse_diff 已用 unidiff 做确定性解析，SCOPE-001/TEST-001/DUP-001 三项核对已是 confidence=1.0 的规则实现——意见 7 的「用确定性工具核对声明」这一半已成立。缺口在：① 解析产物内部消化，未成一等结构，报告里没有给人看的「修改范围概览」；② map_verdict 的 blast_radius 由发现类别（模型来源）推导，安全/契约敏感路径的命中判断依赖模型报出来，而非路径规则直接算出来。

修改：

  • 新增一等结构 ScopeFacts（定义见 §4.1），由 contract_check.scope_facts(diff) 纯规则产出。
  • 新增仓级路径规则文件 .touchstone/scope-rules.yaml：声明契约目录、安全敏感路径、签名相关路径的 glob 列表（human_curated，与 acceptance.yaml 同级待遇）。
  • blast_radius 的推导顺序改为：路径规则命中（确定性）∪ 发现类别推导（模型，补充）。改动只碰到敏感路径但模型没报任何发现时，影响面照样点亮——消除「模型漏报导致影响面漏判」的路径。
  • 报告版面第③段呈现范围事实摘要，人第一眼即可看到本 PR 的机器实测修改范围，与 author 的 SubmissionContract 声明并排对照。
  • ScopeFacts.fingerprint 作为意见 10 指纹的输入（见下）。

意见 10 · 轮次台账：同内容重提刷不出新轮次

现状：轮次以 PR 号记账（LoopState 从该 PR 自己的评论 marker 派生），关旧 PR、原样开新 PR 即重置轮次并让评审方失去历史记忆——这是会上确认的可钻漏洞。

修改：

  • 指纹：fingerprint = (文件集哈希, 每文件 hunk 结构摘要)，由 ScopeFacts 派生。同源判定：文件集 Jaccard ≥ 0.8 且 hunk 结构相似度 ≥ 阈值（两者均为控制变量，进锚定矩阵）。不用全文哈希——改一个空格即可绕过。
  • 不新增存储：台账不建库。PR 打开时（preflight 阶段）经 API 检索近 30 天已关闭 PR，按文件集初筛，再对候选取 diff 算指纹比对；命中即从该关闭 PR 的置顶评论（清单＋marker）重建台账——与现有「状态存于评论 marker」的机制同构，无额外一致性负担。
  • 命中处理是记录＋提示，不是拦截：继承轮次余额与未销项清单，报告横幅明示「与已关闭的 #N 同源，历史已消耗 X 轮、Y 条未销项，本 PR 剩余轮次 Z」；余额为零直接升级人工。正当的关旧开新（rebase 损坏、换分支）不受阻，只是历史欠账原样跟随。
  • 人工重置口：PR 打上 rounds-reset label（需仓库写权限）即显式授权重置台账——例外权归人，与系统「人是最终决策者」的定位一致。
  • 行为信号入学习回路：「轮次耗尽→关闭→同源重提」作为提交方行为层面的负样本素材记入学习回路，它比单条代码问题更能说明提交方在回避收敛。

4. 核心数据结构（新增与修改）

4.1 ScopeFacts · 范围事实 新增

8. 一致性校验

已发现并在本文中解决的冲突：

  • 意见 2 与自改循环的依赖冲突：author_actionable 原以 suggested_fix 为门槛，废弃该字段会使循环空转——已将门槛改为 fix_direction，循环语义不变，确定性来源保留 deterministic_patch 作精确修复通道。
  • 「达成判据」与既有 acceptance.yaml 的概念区分：前者是发现级销项判据（评审方产出），后者是 PR 级验收规格（人核准、驱动验证子系统）。两者层次不同、命名不混用，主文档 §3.1 的表述随之补注。
  • 意见 10 的存储方案与既有「状态存于评论 marker」机制同构：台账从关闭 PR 的置顶评论重建，不引入新存储，也不产生双源一致性问题；防篡改沿用 trusted_bodies 过滤。

有意接受的遗留项：

  • 指纹对「刻意打散文件结构的同内容重提」有绕过空间（大幅改名＋重排 hunk）。接受理由：绕过成本已显著高于收益，且此类行为本身会被行为信号记录；完全防御需语义级比对，代价与定位不符。
  • review 类达成判据的复核仍依赖模型，存在与被复核对象同源的相关性盲点。接受理由：与主文档 §2.2 问题三的既有结论一致——独立性主要来源仍是确定性门禁与验证子系统，复核判据只求把「凭感觉再挑」收敛为「定向回答一个问题」。
  • 意见 8（动态测试分层采样）、意见 9（评分分栏）、意见 11（提交前冒烟门禁）本次明确不采纳，相关接缝（route 的验证档、可插拔检查）保持现状，不做预留性改动。

逐维检查结论：概念一致性——四个新概念全文单一命名，与既有概念无同义异名；状态完备性——清单状态机四态齐备、非法转移有回退策略，台账在「无历史命中／命中／人工重置」三输入下行为均有定义；接口完备性——§3 各修改点在 §5 均有对应接口；层次一致性——范围事实归确定性层、方向与判据归评审层、销项复核归循环层、重置权归人，未越层。
