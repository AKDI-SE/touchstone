TF-GRPO 自进化闭环设计

TF-GRPO 的机制骨架已实现且对齐设计，docs/learning-loop-design.html §3.6 的生产化差距清单是「差距地图」而非「施工图」。本设计把三块设计盲区补成施工级，核心是破冷启动死锁——让 candidate 经验真能自动转 active、自进化闭环真正转起来，同时守住 advisory-only 红线。

设计对象　冷启动死锁破局（详）· 坏真值检测（中）· 运行编排（降级说明）
日期　2026-07-20
依据　精读 experience_store.py / ground_truth.py / learning_loop.py / orchestrator.py / distill.py + learn.yml + audit-tfgrpo.html + learning-loop-design.html §3
产出　设计文档（A-lane）；代码实施按 §2.7 的 6 步另开 PR

定位（与 §3.6 的关系） 本设计是 learning-loop-design.html §3.6 的施工化补充，不替代它：§3.6 第 1 类（奖励质量）与第 5 类（真值清洗）的留白，在此分别由盲区 1（冷启动）和盲区 2（坏真值）给出可施工方案；第 3-4 类（收敛/成本）见 §4 降级说明。骨架机制（I1–I4 / 数据管线 / 门控）已实现且不动，本设计只在数据采集与真值清洗两个侧面拓宽。

01

背景与两个关键修正

评估 TF-GRPO 设计完备性时（详见记忆 tfgrpo-design-completeness-assessment），识别出三个"设计盲区"：冷启动、坏真值、运行编排。深入代码后，两个判断需要修正——这直接决定了下文的设计重点。

修正一 · 运行编排其实已全接通（先前误判为盲区） .github/workflows/learn.yml 每周一 02:00 UTC cron → 跑 calibrate → 跑 learning_loop.main()（L72-201，main 里确调 graduate L170 与 retire L178）→ 经验库若有变化则 commit 回 main。评审与学习解耦（评审走 PR 事件、只读经验库；学习是离线 cron）、持久化（atomic_write_json + 条件 git commit）、真值重建（build_ground_truth）全部到位。→ 盲区 3 降级为"缺收敛/成本控制（§3.6 第 3-4 类施工）"，不是编排缺失。
修正二 · 冷启动死锁确认真实，且比 audit D2 更深一层 render_injection（experience_store.py:204）只注入 status=="active"；candidate 从未被注入 → 历史 PR 的 injected_types 不含 candidate 的 type → aggregate_ab（ground_truth.py:74）对该 type 的 with 臂恒为 0 → graduate（experience_store.py:138，ws<20）永远跳过。接通数据管线（audit D2 已修）≠ 打破死锁——TF-GRPO 蒸出的 candidate 永远卡在池里，唯一进注入的是人手写 seed_experience（直接 active+locked，experience_store.py:88），那不算"自进化"。

因此本设计的重心是盲区 1（冷启动死锁破局），给出详案；盲区 2 给中案；盲区 3 给降级说明。

02

盲区 1 · 冷启动死锁破局（详案）

2.1 死锁机制（已逐行核实）

  • render_injection（experience_store.py:200-210）只取 status=="active"，经 _resolve_conflicts（I3）后渲染成 PR-Agent 的 extra_instructions。
  • graduate（experience_store.py:125-146）：candidate→active 需 A/B 两臂各 ≥ GRADUATE_MIN_SAMPLES=20（L23）且 lift = (wa/ws) − (oa/os) ≥ 0.10（L24）。
  • aggregate_ab（ground_truth.py:59-82）：with 臂 = 该 finding_type 在历史 PR 的 injected_types 里出现过（即评审时是 active 被注入过）；without 臂 = 没注入过。
  • injected_types 由评审时 orchestrator 经 active_types(store) 写进 <!-- touchstone-result: --> marker。
因果链candidate 从未 active → 任何历史 PR 的 injected_types 都不含 candidate 的 type → aggregate_ab 对 candidate type 的 with_seen 恒 0 → graduate 恒跳过。ground_truth.py:64-65 的注释自己也承认这一冻结态："冷启动期 with 臂为 0……需先靠 seed 注入积累。"——但 seed 是人写的（source="human"），不是自进化。

2.3 推荐：(a) + (c) 组合，否决 (b)

否决 (b) 的根本理由"无 A/B 证据即激活"在数据层放宽了门控断言——违反 experience_store.py:124 的设计纪律（"candidate 先入池、经 shadow A/B 达标才转 active"，即坑 3）。retire 兜底是"事后纠错"，但 RETIRE_MIN_FIRES=8 意味着坏经验至少误导 8 个 PR 才退役，advisory-only 红线下风险敞口不可接受；且 counting 蒸出的 candidate 根本没有 group_rewards，适用面窄。
为什么是 (a)(a) 是唯一真正"破"死锁的方案：让 candidate 的 with 臂从 0 真实增长到 ≥20，graduate 的 lift 判定全流程不变，A/B 门控纪律完全保留——它只在"数据采集"侧拓宽一个口子（candidate 也参与采数），不改"激活判定"侧的任何阈值。
为什么加 (c) 作辅助纯冷启动（经验库空、calib_agg 稀疏）时，shadow 注入要"该 type 在 raised_types 里出现"才有意义，若该 type 极少被挑，with 臂积累极慢。(c) 的 bootstrap seed 让该 type 至少有一条 active 经验、立即进 active_types，使后续同 type candidate 的 with/without 对照有稳定基准、评审质量先受益。但 (c) 不能替代 (a)——它只把"第一个 active"从人手变自动，candidate 自己仍走正常 graduate。

2.4 实现路径（函数签名级，A-lane 范围）

Experience 结构（不改现有字段语义，仅扩，契约安全）：

  • status 枚举 candidate | active | retired → 加 shadow（语义="采 A/B 数据、隔离标灰注入、未达门槛"）。现有三值语义不变 → 现有测试不受影响。
  • evidence 加 shadow_fires（采数 PR 数）、graduated_via（"ab" | "bootstrap"）——向后兼容，缺失即默认。
  • source 加 bootstrap（仅 (c) 用）；seed_experience 的 source="human" 不变（契约：seed 是人写的）。

配置开关（env，默认全关 = 字节级零行为变化，这是回退保险）：

2.5 风险与缓解

采数期影响（最大风险）candidate 在 shadow 期进入 extra_instructions，会真实影响 PR-Agent 的建议。若 candidate 是错的：suppress 类误注入会降召回（缓解：SHADOW_MIN_EVIDENCE 初筛 + protected_types 强约束 + 文本标灰）；emphasize 类误注入会增噪声（缓解：candidate 文本来自旗舰模型内省、非用户输入，投毒面小）。advisory-only 保证不进 contract_check / verify / 总闸——坏经验最坏让一条建议偏一点，人忽略即可。
确定性抽样（必须）必须用稳定哈希（hashlib，非 hash()），否则同 store 在不同评审进程选不同 shadow 集 → 归因混乱。
时序安全（已核实）marker 是评审时快照，build_ground_truth 读历史 marker，与 candidate 当前 status 无关——candidate 在 PR#N 评审时是 shadow、之后才转 active，归因仍正确落到"曾 shadow 注入"。
bootstrap 过拟合冷启动期 calib_agg 样本少，某 type 偶然全采纳可能触发 bootstrap。缓解：MIN_FIRES=15（高于入池门槛 8）抑制偶然高采纳。

2.6 冷启动预估（基于 min_samples=20 / GT_WINDOW=30）

  • 纯 (a)：某 type 每 30-PR 窗口 raised ~10 次、shadow 比例 0.5 → 每周期 ~5 个 with 臂样本 → 首个 candidate active 约 4-5 周期（120-150 PR 流量）。
  • (a) + (c)：bootstrap seed 立即产 1 条 active emphasize → 该 type 立即进 active_types → 评审质量先受益；candidate 自己 active 仍 4-5 周期。

说明：4-5 周期是以"默认 G=4、GT_WINDOW=30、shadow_ratio=0.5"估算。可调 TOUCHSTONE_SHADOW_RATIO=1.0（牺牲安全裕度换速度）压缩到 2-3 周期。

2.7 可独立合入的 6 步（每步默认关 env、现有测试全绿、可回退）

  1. status=shadow + evidence 字段 + shadow_candidates + render_injection(include_shadow=False)（默认关）。
  2. aggregate_ab 拓宽 with 臂 + make_gt_entry 加参数（向后兼容）。
  3. orchestrator marker 加 shadow 字段（先确认 _parse_result 容错）。
  4. review_provider include_shadow 透传 + 受信 ref 同等约束。
  5. 验收测试 test_cold_start_candidate_graduates_via_shadow（先红后绿——破死锁的可证伪锚点）。
  6. （可选）bootstrap_from_calibrate + main 调用点。

03

盲区 2 · 坏真值检测（中案，冷启动之后）

3.1 现状（已较完善）

采纳判定（calibrate.py + ground_truth.py）已覆盖：isResolved + wontfix 关键词过滤（_DISMISS L78-87）+ 作者自 resolve 排除（L98）+ 非 trusted-author marker 跳过（L101）。硬过滤 2 条（无 marker / 取数失败），软降权 3 条。

3.2 真实留白与设计

留白 1 · reviewer 分级加权现有 GraphQL（calibrate.py:45-54 _GQL_THREADS）不查 author_association。设计：扩查询拿 comments.author.association（OWNER / MEMBER / COLLABORATOR / CONTRIBUTOR），低权重 reviewer 的采纳信号软降权（不剔除）。
留白 2 · LGTM-only 识别现在只取 review state，不看 body / 评论数。设计：识别 review body 简短（<N 字）或仅含 LGTM/Approved 关键词的"一键过" PR，软降权。
留白 3 · 坏真值检测机制（§3.6 第 5 类给的是循环引用，须给可观测信号） 信号 A：某 PR 进真值集后蒸馏时全组 reward 系统性低（human_adopted 与 raised 几乎不重叠）→ 真值可疑。数据基础：evidence.group_rewards + source_prs（部分有）；缺 PR-reward 历史序列，设计需补持久化。
信号 B：LGTM-only（见留白 2）。
信号 C：低权重 reviewer（见留白 1）。
信号 D：改动行数极少却 resolved（GT_DIFF_BUDGET 已取 diff，可复用统计）。
处理策略：默认软降权（进真值集但 weight 低），仅多信号叠加且 reward 离群时硬剔除。
定位盲区 2 是奖励质量（§3.6 第 1 类）的前置——坏真值直接污染 reward → 蒸馏出低质量经验。建议冷启动之后、奖励质量之前做：冷启动让闭环转起来，坏真值清洗让转起来的信号更干净。
