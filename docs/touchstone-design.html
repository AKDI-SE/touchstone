三类判定评审在回答三种问题：做对了吗（正确性，能跑验证）、越界了吗（红线/契约，能写成规则）、做好了吗（质量，没有标准答案、只能判断）。本设计特有的分法；三类各近似：功能/正确性测试、合规即代码（policy-as-code）、代码质量与可维护性。
§ 1

设计目的

现状流程是这样的：AI agent 大量产出代码并提交 PR，再由人逐个审核。瓶颈出在人这一环——AI 写得又快又多，人没有足够精力逐行读完每一个 PR。本设计在这条已有流程上新增一个 Touchstone，不重写流程、也不从零造系统：它是一道自动评审层，位于 PR 提交之后、人决定是否合入之前，对每个 PR 给出评分与反馈。

它的输出有两个使用者。一是写代码的 agent：拿到反馈后自行修改、重新提交，形成机器对机器的快速迭代。二是人：把评分和反馈当作参考，据此决定某个 PR 值不值得看、要投入多少精力。本设计的目标随之确定——把人从逐个 PR、逐行通读的负担中解放出来，理想情况下让 Touchstone 在尽可能多的 PR 上代替人工审核，同时不让有缺陷的代码因为无人细看而被合入主干（即不以牺牲质量为代价）。

这个目标不是无条件就能达到的。要让方案站得住，有三件事必须先讲清楚。

  • 第一，评审能否替人把关，最终取决于代码有没有被真正运行、验证过，而不是 AI 读得多仔细。但对每一个 PR 都跑完整的后端集成验证，成本无法承受。这两者直接冲突：验证做得越少，能放心交给 Touchstone、无需人工复核的 PR 就越少。本设计不回避这一冲突，而是把它转化为一个可以逐步推进的指标——验证边界：客观验证能覆盖到哪里，自动化就推进到哪里。
  • 第二，代码是否正确，仅凭阅读无法判断。AI 代码最危险的一类缺陷，是看上去正确、也能运行，但逻辑其实有误——本文称之为似是而非的错误，这类问题不真正执行就暴露不出来。因此在正确性方面，Touchstone 只能帮助人更高效地定位该重点检查的地方，无法靠纯阅读把人的工作量完全替代。
  • 第三，Touchstone 现阶段只给出参考意见，与人工审核并行，并不拦截合入。这是一个低风险的起点。在此基础上，本设计规划了一条演进路线：自主权逐步放开，但始终以真实验证为托底，而不是一开始就把合入的决定权交给 AI。
一句话概括 用确定性的机制守住不能出错的部分，用 AI 的判断力改进可以做得更好的部分，用真实运行结果回头检验 AI 的判断本身是否可靠，让人专注于定义什么才算正确。速度交给 agent，底线交给验证，判断力交给 Touchstone，方向盘和校准权留在可审计的客观结果与人手里。
§ 2

设计逻辑与概念空间

本节先建立贯穿全文的概念空间，再围绕三个核心问题说明设计选择与取舍，最后给出四个逻辑视图（职责分层、PR 流程、风险分流、演进边界）。注：默认形态是顾问式评审、人点合并。本节中凡涉及"自动合并"的概念与论证（验证边界、自动放行等），界定的都是（可选、默认关）的进阶路径，不是默认形态。

2.1 概念空间

下列概念都不是预先拍脑袋定义的，而是为解决后面每个具体问题而引入；后续的数据结构与接口必须直接使用这些名称。

三类判定
把评审要回答的问题拆成三类，可判定性各不相同：正确性可执行验证、红线/契约可规则化、质量无客观真值需判断。
验证边界 Verification Frontier
合入决策能完全靠客观验证站住的那部分 PR——它界定（可选的）自动合并最多能覆盖到哪。默认形态（人点合并）不涉及它，仅当开启自动合并时才是约束。
Touchstone
评审 agent。其角色是编排验证 + 质疑 + 低风险质量判断，不是预言机，其评价不构成准入依据。
风险分流（Risk-Based）
Touchstone 产出的一个风险信号，同时把两样稀缺资源分流到不同处理路径：人的注意力，与验证预算。
提交契约 Submission Contract
强制 写代码的 agent 在提交 PR 时随附的结构化信息（意图、验收标准、复用项、自标风险、重点看哪里）。
只用于定位、绝不采信
治理提交契约的铁律：契约只用于定位（指出该重点看哪里），绝不被采信；Touchstone 一律独立核对，对不上即问题。
变更比例验证
不跑整套集成，而是针对本次改动现生成少量针对性测试、只跑这些（隔离、快），让验证量正比于改动量。
单一事实源规范
一份规范，同时提供给 author（写对，把问题前移到写代码阶段）与 Touchstone（查错）；可机器自动核对 的条目可被固化为确定性检查。
把“建议”升级为“拦截门禁”的通道
反复出现且被证实有害的 第三类 发现，被固化进确定性门禁、升级为可拦截的 第二类 门禁——这是建议最终升级为拦截门禁的唯一通道。
校准（Calibration · 影子模式下）
在影子模式下（与人审并行、不影响合入），拿人审的真实裁决当标准答案，持续测量 Touchstone 与人的吻合度。
反馈循环（Feedback Loop）
写代码的 agent 拿到反馈后改代码、再重新提交；机器间快循环，设轮次上限，且其产物在正确性上仍需独立仲裁。
熔断 Circuit Breaker
用合入后的 revert/hotfix 率与 Touchstone 批准率漂移做监控；越阈值则自动收回自主权、回落到人。
评审引擎（PR-Agent）
复用开源 PR-Agent 产出评审观察与建议（多工具、建议去重/排序、多语言）。评审层的权威设计见配套文档 touchstone-on-pr-agent.html。
发现归一 + 裁决映射
normalize 把 PR-Agent 输出归一成本系统 Finding；map_verdict 按 category 定单一 RiskAssessment（不做共识——去重/排序已由 PR-Agent 完成）。
范围事实 ScopeFacts
确定性工具直接从 diff 计算出的修改范围事实（每文件增删/hunk 结构、敏感路径命中、内容指纹），不依赖任何声明、不经过任何模型——继「硬门禁结果」「经验知识库」之后的第三个客观锚。敏感路径按仓级 .touchstone/scope-rules.yaml 规则命中，直接点亮影响面（模型漏报不再导致影响面漏判）。
达成判据 done criteria
每条评审发现附带的、可核验的「改到什么状态算过关」描述。确定性判据优先（规则复检不再命中）；给不出时给复核判据（一句定向复核问题）。author 知道终点在哪，销项有依据。
收敛清单 ConvergenceChecklist
双 agent 交互围绕的逐项销项清单（方向+依据+达成判据）：置顶评论 task list（人可读）+ 隐藏权威 marker（机器可读，只信机器人评论）双载体。author 经 touchstone-ack 申报——申报只是输入信号，评审方按达成判据复核后才销项。清单销项率取代「无新发现」成为收敛指标。
轮次台账 RoundLedger
轮次记账主体从 PR 号改为内容指纹（由范围事实派生，文件集 Jaccard + hunk 结构双阈值）。同源的「关旧开新」继承历史轮次消耗与未销项清单（从关闭 PR 评论重建，零新增存储），重提刷不出新额度；rounds-reset label 人工授权重置。

2.2 核心问题与取舍

问题一：AI 判断该承担什么角色——它给建议，绝不当合入的准入门禁

最直接的想法，是做一个像资深工程师那样通读 diff、再判断这个 PR 能不能合入的评审者。本设计不采用这个方案：它仍然把合入与否押在判断之上，只是把判断者从可靠的一方换成了不可靠的一方。人这一方是可靠的——拿不准时会明说，背后还有验证和回滚保障。AI 这一方则不然：它会自信地批准那些似是而非的错误；而且评审用的 AI 往往和写代码的 AI 是同一类模型，author 看漏的地方它多半也看漏，两者共享同一批盲区；再加上人一旦退出，就没有东西能兜住它的误判。

因此，AI 的判断不进入合入的准入环节。默认形态下，合入与否始终由人决定，Touchstone 只产出评审意见；是否设置一道必须通过才能合入的硬性关卡，交给可插拔的质量门禁（见 §4.7）——由现成的规则工具与确定性检查保障，而不依赖 Touchstone 的判断。即便（可选地）开启自动合并，放行的依据也只是客观验证的结果，而非 Touchstone 的判断。由此，评审要回答的问题被分成三类，各自交给不同的决策者：

问题二：付不起每个 PR 跑集成验证，如何既减人力又不失正确性

两个极端都不行：每 PR 全集成验证（太贵，被你否掉）；纯阅读、零验证（读不出正确性）。采用的折中是把人的工作量先一刀切成两层，再用一个统一信号去分配稀缺资源：

  • 结构层（重复造轮子、违反约定、过度工程、漏的错误处理、恒真测试、越界改动）——读 + 廉价静态信号即可抓，交给 反馈循环，人基本不看。这是人力下降的大头。
  • 正确性层——Touchstone 与 author 都读不出，只能靠验证。用 风险分流 把稀缺的验证预算与人的注意力只导向高风险的少数：低风险走廉价信号 + 反馈循环、人不看；高风险才跑 变更比例验证 并送人细看。
关键洞察 付不起完整验证，不等于没有任何客观信号。lint、类型检查、快单测、SAST 都不是集成验证、都很便宜，照提供给 Touchstone 当依据。要省的是 端到端/集成那层；把它从每个 PR 必跑，降为只对风险分流标高的少数 PR 跑，就同时解决了成本与正确性保障。

问题三：author 与 Touchstone 同源——反馈循环会不会变成互相放松把关

否决还有一个错误假设是相信反馈循环会自然收敛到正确：同源模型一起瞎，author 会把代码改成「让 Touchstone 满意」而非「真的更对」，闭环能把一坨东西抛得更亮、却抛不出正确性。采用三道约束：

  • 独立性，按强弱排序——最硬最持久的是对独立测试执行（由 verify 子系统承担，可选）；其次是信息不对称，即让 Touchstone 掌握 author 生成时未必有的信息——目前主要是人核准的验收规格（走 verify 侧）；最弱、且随模型趋同越来越没用的是换个模型。默认形态下真正承重的是对独立测试执行那条，换模型仅作可选点缀。
  • 轮次上限 + 升级判据：闭环设最大轮数，防震荡，也防止双方妥协着把标准降下来；超限或反复矛盾则升级到人或 re-plan。
  • 闭环产物不自证正确：其正确性仍由变更比例验证或人来仲裁。
关于模型多样性 让 author 与审核侧用不同模型——做，它便宜、有真实但有限的去相关收益，且最该用在反馈循环里（同源模型会收敛到它自己觉得顺眼的写法这一不动点，而那可能正好是错的）。但要清醒：它是三种独立性来源里最弱、且随前沿模型趋同在衰减的一条，不能当主力；它不跨越验证边界（两个模型来读，终究还是只是读）；审核模型不得弱于 author（逮错比制造错更难，挑弱模型是假经济）。最值钱的形态是“不同模型 + 对抗角色 + 信息不对称”三件套，而非单换模型。

§ 3

核心数据结构

只定义与设计逻辑对应的关键成员，不写编码细节。七个结构对应概念空间的七个要点。

3.1 SubmissionContract · 提交契约

4.4 固化与熔断

# 把“建议”升级为“拦截门禁”的通道：把反复出现且证实有害的发现升级为确定性门禁
promote_to_gate(calibration_agg, rule_index)
    # 候选条件：可机器自动核对 AND 复发(fires≥N) AND 高采纳(人改动比例≥阈值,
    #   作为"证实有害"的可靠归因代理——与噪声检测正好相反)
    # 效果：rule.enforced=true(确定性门禁)。改 SoT 须以（提案）交人确认,不静默改

# 熔断：按真实结果收回或给予自主权，替代「人工把关」这道防线
update_autonomy(merge_records, prior_approval_rate)
    # revert/hotfix：用可靠归因(git revert 检测 + merge_commit_sha 匹配),低体量下干净够用
    # 越阈值 → tripped：收回自主、回落到人；Touchstone 低风险比例突升 → 告警(疑似把关变松/被钻空子)

关系这两个接口让循环长期保持"正"：promote_to_gate() 是"建议"变"门禁"的唯一通道，让质量问题最终可被强制执行；update_autonomy() 用现实结果当否决权，确保即便 Touchstone 被钻空子，自主权也会自动收缩。

4.6 渐进开放自动合并（可选，默认关）：执行器 + 经验层

这是（可选）路径，默认关、需显式开启；不是默认形态——默认是顾问式评审、人点合并。本节描述：当你选择开启自动合并时，它如何只在安全前提下替人点合并。第一原则不变：评审可能出错，授权自动放行的从来不是评审意见，而是汇总状态检查（质量门禁）的通过。下方判据多道门禁串联、默认全关、经验门控、熔断保障。（已实现：判据已简化为"直接读汇总状态检查(统一门禁)是否绿"，与 §4.7 的质量门禁对齐；契约/规则/可选 verify 均折进统一门禁，autonomy 不再自行判定地基。）

自动放行判据(decide_auto_merge)——当且仅当以下全部成立，否则回落到人：

1 质量门禁通过   汇总状态检查(统一门禁)= success。契约核对、确定性规则、(可选)verify 都已折进统一门禁；autonomy 只读这一个结论，不再自行判定地基。（统一门禁非 success 即不放行）
2 阻断否决       无未消除的 block_candidate 且非 high 档。（不对称：阻断条件能否决自动放行，但不能批准——批准只由质量门禁给）
3 闭环已收敛     loop_decision=converged（非 escalate）
4 未熔断         govern.update_autonomy 维持（非"收回自主"）
5 变更分类已达标 经验层判该类变更有良好的安全记录（见下）
6 base 未过期    合并目标分支自评审后未前进（base_fresh，防合并偏斜——评的是 A 合的是 A+Δ）
7 评审可信       review_reliable=true（LLM 空响应/引擎降级轮不放行，防"0 发现"假绿）
8 无自证销项     unverified_claims=0（waived/split 系 author 自证、机器未核实，见阶段十一）
9 作者受信       author_trusted（GitHub author_association ∈ OWNER/MEMBER/COLLABORATOR
                 或显式 login 白名单；产物缺作者出处即 fail-closed 不放行，见阶段十三）
10 总开关开      AUTONOMY_ENABLED（默认关；AUTONOMY_SHADOW 影子模式只记不合）
变更分类auto-merge 的资格按（变更分类）累积与授予，不按单 PR。签名 = {风险等级 · 文件类型(docs_only/test_only/code/mixed) · 触及的规则 category 集 · blast}。够粗以积累足够样本、够细以有意义。
怎样安全地分级开放难点：某类变更要先有足够样本才敢放开，但不放开就拿不到样本。做法分三步：① 影子模式(AUTONOMY_SHADOW)——对人工合并的 PR 只记录"按规则本会自动合并吗 + 它后来是否被回滚/热修"，只观察、不自动合并；② 达标——某类变更累计样本≥N 且回滚/热修率≤阈值，才取得自动合并资格；③ 开启——总开关打开后，对已达标的变更分类才真正自动合并。先观察、后达标、再开启——不对没有先例的变更分类冒险。
经验层与达标经验库 = 按变更分类的 {样本数, revert, hotfix, 人否决}，由 merge_records 累积(沿用 calibrate/govern 管线)。graduate_classes：样本≥N ∧ 坏结局率≤阈值 → 达标。与 §4.4 的 promote_to_gate(规则固化)同构——只是这里固化的是"自动放行某类变更"的资格。TF-GRPO 是把这套从计数式达标升级为可增删改经验库(把手写 standards 学成经验)的演进路径。
经验层不替代任何单 PR 门禁即便类已达标，每个 PR 仍须逐条过 1–10 道门禁。经验层只决定"这类变更我们是否愿意自动放行"，绝不替代某 PR 的质量门禁/否决/收敛/熔断。
与熔断的关系经验层是前向的按类达标门；update_autonomy 是全局熔断后备(自动放行后 revert 率超阈 → 收回自主、强制关)。二者都吃 merge_records，一前一后。
GitHub 形态job 链：touchstone(评审，TOUCHSTONE_SKIP_GATE 不自发统一门禁) → (可选)verify → gate(聚合并发那一个统一门禁、写回 touchstone-findings.json) → auto_merge。auto_merge job needs [touchstone, gate]，默认不启用(if: vars.AUTONOMY_ENABLED=='true')：读 gate job 写回的统一门禁结论(touchstone-findings.json 的 gate，已含 verify) + 达标类(由 govern 定时任务发布) + 熔断态 → decide_auto_merge；merge=true 则 execute_auto_merge 调 merge API 并打 auto_handled marker 供校准归因。
边界（默认放行 0 个）默认 AUTONOMY_ENABLED 关 → 放行 0；经验为空 → 无类达标 → 放行 0。要真放行须同时：类达标 ∧ 总开关 ∧ 质量门禁(统一门禁)绿 ∧ 阻断否决无 ∧ 闭环收敛 ∧ 未熔断。任一不成立即回落到人——这是（可选）自动合并的安全前提：开启后也只在质量门禁绿等条件全满足时才替人点。

4.7 可插拔检查门禁（默认形态对外的硬质量门禁）

这道门禁是 §2.2 问题一里“必须通过才能合”的硬质量门禁的机制实现，§4.5（GitHub 集成）、§4.6（autonomy）、§6（verify）都用到它。Touchstone 自己不发明关卡，只提供一个挂载点——把现成质量工具（或以后自建的服务）的结果挂成检查，再按 .touchstone/checks.yaml 里的策略汇总成一个对外状态。改“挂哪些、哪几个必须通过”只改这个文件，不碰 GitHub 设置。

一个检查 = 名字 + 通过与否 + 一句话摘要

框架读配置 → 跑所有启用的检查 → 把每个结果折进汇总状态检查。三种挂法：

builtin
进程内函数。如 touchstone-rules（自带确定性规则，无拦截级才通过）、verify（折入按需深检结果）。
relay
读某个已有 GitHub check-run 的结论——工具在它自己的 CI 里跑过（单测 / SonarQube / SAST），这里只转达。
service
POST PR 上下文到一个 HTTP 服务拿结果——未来自建质量服务按此接口挂上来，不改框架。
汇总状态检查 aggregate_gate
所有 required 检查都通过才 success；任一必须项非通过 → failure。非必须项只作信息。防静默故障：① checks.yaml 存在但解析失败（区别于未配置）→ 总闸 failure 并在 summary 报警，不当空策略放行；② gate job 拿不到 touchstone-findings.json（touchstone job 失败/取消）→ 用 head sha 发一个 failure check-run 说明"评审流水线未产出结果"，不静默 no-op。
与 GitHub 的最小耦合对外只发一个汇总状态检查(默认 touchstone/gate)——由专门的 gate 步骤在 Touchstone 评审与(可选)verify 之后聚合所有启用的检查、统一发布这一个状态。人点合并场景：GitHub 分支保护一次性只要求这一个状态即可——挂几个插件、哪几个必须绿、阈值多少，全在 .touchstone/checks.yaml 里演进，不再回去动 GitHub。自动合并场景(可选)：Touchstone 自己看汇总状态检查绿不绿决定合不合，零 GitHub 配置。

§ 5

一致性校验

对全文设计要素做综合校验。通过的维度一笔带过，重点记录发现并修正的问题、以及有意接受的遗留项与理由。

通过的维度

  • 概念一致性：三类判定、验证边界、风险分流、提交契约、变更比例验证、把“建议”升级为“拦截门禁”的通道、校准、反馈循环、熔断、单一事实源规范、评审引擎（PR-Agent）、发现归一、裁决映射，以及验证子系统的独立验收测试、职责分离约束、充分性校验、空实现探针（把功能改空看测试会不会报警）、变异验证等概念全文同名同义，数据结构与接口直接复用其名。
  • 接口完备性：设计逻辑中每个操作均有对应接口（评审→review_pr；分流→route；验证→verify_change；闭环→loop_step；校准→record_calibration；固化→promote_to_gate；熔断→update_autonomy）。
  • 评审意见落实（2026-07-04）中发现并解决的冲突：① 意见 2 与自改循环的依赖冲突——author_actionable 原以 suggested_fix（补丁）为门槛，废弃该字段会使循环空转；已改为 fix_direction 门槛，确定性来源保留 deterministic_patch 精确修复通道。② 「达成判据」与既有 acceptance.yaml 的概念区分——前者是发现级销项判据（评审方产出），后者是 PR 级验收规格（人核准、驱动验证子系统），层次不同、命名不混用。③ 台账存储与既有「状态存于评论 marker」机制同构——从关闭 PR 的机器人评论重建，不引入新存储、无双源一致性问题，防篡改沿用 trusted_bodies。
  • 有意接受的遗留项（意见落实）：指纹对「刻意大幅改名+重排 hunk 的同内容重提」有绕过空间——绕过成本已高于收益且行为本身被 lineage 记录，语义级比对的代价与 advisory 定位不符；review 类达成判据的复核仍依赖模型（与 §2.2 问题三既有结论一致，独立性主要来源仍是确定性门禁与验证子系统）；意见 8（动态测试分层采样）、9（评分分栏）、11（提交前冒烟门禁）明确不采纳，相关接缝保持现状、不做预留性改动。
  • 层次一致性：Touchstone 不承担质量门禁的职责——它从不对正确性给出准入裁决，准入只由确定性门禁产生；Touchstone 的输出对正确性只构成风险分流与建议。

PR 评论中出现伪造的 loop markerloop 状态只从【机器人自己】发的评论解析（按发帖人过滤，loop.trusted_bodies）；身份无法确认时降级为全量并告警
Touchstone 批准率突升熔断告警，疑似把关变松/被钻空子
合入后事故率越阈值熔断收回自主权，全部回落到人

有意接受的遗留项

  • 第三类 质量在全自主下会缓慢漂移（无客观门禁）。接受，并由风险分流圈住：仅在架构影响低处全自主，触及公共 API/schema/契约/安全面的少数保留人。
  • author 与 Touchstone 同源的相关性盲点不能根除，只能缓解（独立性三来源 + 闭环产物不自证正确）。接受。
  • 当前校准信号继承人审盲区：以「与人一致」为奖励，会连人看不出的 似是而非的错误 一起学过去。接受为 Phase 1 限制；Phase 2 加入变更比例验证的客观结果作为更高质量奖励后改善。
  • 人力下降在正确性轴上有上限：上限 = 可负担的廉价验证覆盖率；结构轴大体可压缩。这是「正确性只能靠跑验证、而验证有成本」的直接推论，非缺陷。
  • 模型多样性的去相关收益有限、且在衰减。让评审侧与 author 用不同模型能带来一些独立性，但这条收益会随前沿模型趋同而衰减，且它不跨越验证边界（两个模型读代码仍然只是"读"）。因此本设计不把模型多样性当作主力独立性来源；评审复用 PR-Agent（单一配置模型），评审侧所用模型要求不弱于 author 所用模型。
  • 评审层只改判断层。评审层（复用 PR-Agent）只能扩展评审广度、降低相关性盲点，而不会新增"客观验证"这一能力。因此（可选的）自动合并的天花板，始终由验证覆盖率决定，而不由评审 agent 的数量或模型多样性决定。
  • 充分性 ≠ 测试意图正确。哨兵与变异只能逮出弱测试，逮不住「强而错」的测试（规格被误读却写得很强）。接受：由人核准的验收规格从源头压低误读、变异部分覆盖、事后熔断保障——充分性层只保证「测了改动」、不保证「测的是对的性质」。
  • 风险分流的安全性部分压在判断层。「Touchstone 全错也越不过门禁」只对准入门禁成立（评审层不能批准合入）；但一个 PR 跑哪一档验证由（会误判的）评审层决定，被误判为 low 的错改只走 cheap_only，自动合并下可能仅凭 CI 绿放行。缓解而非根除：map_verdict 除按类别定级外，再从改动文件路径确定性计算 blast_radius（review_provider.deterministic_blast：migration/*.sql/*.proto/schema → cross_module_contract；auth/crypto/secrets 等路径 → security_surface），与评审侧结果保守取并；一旦命中严重影响面，无视 LLM 类别直接抬到 high → full_suite，并触发（可选的）自动合并否决。即评审侧漏判类别时，路径证据仍把伤害面大的改动拦在自动放行之外；再由事后熔断兜底。
§ 6

验证子系统：独立测试来源与充分性

本节描述的 verify 是（可选）的深度正确性检查（默认关、算力够时人按需开），不是默认形态的一部分。它作为一个检查插件挂进质量门禁（见 §4.7）：开启后产出一个可信的"正确性验证"结果。本节讲的就是——这个结果"凭什么可信"。验证 = 跑测试；若测试不可信（author 自写恒真测试）或不充分（跑了却没真正测到改动），它给出的就是假绿。锚点同主文档（2026-06-18 / GitHub PR）。

6.1 子系统目的

定性。这是 verify 插件的内部细化设计。目的：让它产出的「验证通过」信号，对本次改动的范围而言，真的等价于「正确」。它要同时解决两件性质不同的事：

  • 来源（Provenance）：验收测试从哪里来，才能不被写代码的 agent 钻空子。
  • 充分性（Adequacy）：即便来源干净，这些测试是否真的测到了改动——还是「在空实现上也能过」的摆设。

这两件做扎实，verify 给出的"绿"才可信——既是给人看的客观正确性证据，也是（可选）自动合并能依赖的前提。

6.2 设计逻辑与概念空间

独立验收测试 Spec-Blind Test
由一个看不到实现、只读 acceptance_criteria 与接口的独立测试作者（异模型）从规格生成的验收测试——编码「规格要什么」，而非「代码做了什么」。
职责分离约束 Separation-of-Duties
author 与 Touchstone 对验收测试集无写权限——不能修改、改弱、跳过那个评判自己的东西。
充分性校验 Adequacy Ladder
廉价→贵、由风险分流分配的三档充分性校验：改动覆盖 → 空实现探针（把功能改空看测试会不会报警） → 变异验证。
空实现探针（把功能改空看测试会不会报警） Empty-Impl Sentinel
廉价底线：同一组验收测试，对改动前/桩实现必须 FAIL、对改动后必须 PASS；否则即恒真或没测到改动，判测试不合格。
变异验证 Mutation Check
强信号、贵、仅高风险跑：对改动区注入小故障，测试杀不掉变异即视为不充分。
改动覆盖 Changed-Line Coverage
廉价必过：测试必须真的执行到改动行。必要非充分。

问题一：来源——谁写验收测试，怎么防 author 污染

author 自写测试，是 specification gaming 的同一向量换了位置：author 既写代码又写测试，二者共享同一份对规格的误解（甚至写恒真测试），于是测试全过、代码却错。否决两个极端：纯人工写全部验收测试（违背减人力目标，不可扩展）；author 自写作为唯一门禁（可被钻空子）。采用：

  • 验收测试主要由 独立验收测试 作者生成——独立 agent、异模型，只看 acceptance_criteria 与接口、看不到实现 diff。它编码的是规格要求，而不是代码现状。
  • 叠加回归套件（仓库已有测试，防退化）与人工关键路径测试（人对高风险路径手工把关，对应人事前定义可验证规格的新角色）。
  • author 自写测试仅作补充信号，绝非唯一门禁；并立 职责分离约束：author 与 Touchstone 对验收测试集无写权限。

问题二：充分性——「通过」是否真的等于「测到了」

来源干净也不够：测试可能很弱，「在空实现上也能过的测试价值为零」。否决对每个 PR 都跑完整变异测试（太贵）。采用 充分性校验，三档由风险分流分配、且验证量正比于改动（接 §4.2 变更比例验证）：

  1. 改动覆盖（廉价，全跑）：测试必须执行到改动行。
  2. 空实现探针（把功能改空看测试会不会报警）（廉价，全跑）：验收测试对改动前/桩实现必须 FAIL、对改动后必须 PASS。这一步用极低成本就杀掉恒真测试与「没测到改动」的测试。
  3. 变异验证（贵，仅高风险）：对改动区注入小故障，测试若杀不掉则判不充分。
防御纵深及其边界 来源（独立性）与充分性（是否真测）互为后盾，但边界要讲清。充分性层（哨兵 / 变异）能逮出弱测试——恒真的、没真正覆盖改动的；职责分离约束保证没人能事后把测试改弱来骗过它。但它逮不住「强而错」的测试：若独立验收测试作者误读了规格、写出强但与意图不符的测试，它会过哨兵、也会在（同样按错误理解写的）新代码上 PASS，形成自信的假绿。所以充分性层保证的是「测了改动」，不是「测的是对的性质」；后者只能靠人核准规格（把规格被误读从源头压低）+ 事后熔断保障。

6.5 一致性校验（本子系统）

  • 补洞：「验证通过」不以 author 自写测试的善意为前提：验收测试来源独立、测试作者与代码作者分离，并由充分性（哨兵 / 变异）保障。
  • 层次一致性：测试作者 ≠ 代码作者；Touchstone 对测试集无写权——评判者与被评判者分离。
  • 有意接受的遗留：① 规格本身的正确性是人的职责——独立验收测试只能让测试忠于规格，规格错则测试错；本子系统建立在「人事前写对 acceptance_criteria」之上（即人被前移的角色）。② 变异验证贵且可能有等价变异噪声 → 风险门控、不全量跑。③ 规格未预见的 unknown-unknowns 任何测试体系都覆盖不到 → 由熔断（合入后 revert/hotfix）保障。
